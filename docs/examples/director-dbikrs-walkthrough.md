# Walkthrough: Директор ДБиКРС («Озенмунайгаз») — реальный документ через реального агента

ФАЗА 7 итогового аудита. Это пример для калибровки экспертов, **не автотест** —
реальные документы и ответы LLM меняются, поэтому здесь нет жёсткого ассерта на
точный балл. Прогон воспроизводим (см. «Как повторить» в конце).

- Источник: `1. ОД Директора ДБиКРС.docx` (Hay Group, © 2015, шаблон КМГ).
- Должность: Директор департамента бурения и капитального ремонта скважин,
  АО «Озенмунайгаз».
- Агент: **реальный** `GroqAgent` (`jeval_agent_provider=groq`, модель
  `llama-3.3-70b-versatile`), не `FakeAgent` — вызван напрямую через
  `JobEvaluator(agent=GroqAgent())`, как в проде при `JEVAL_AGENT_PROVIDER=groq`.
- Версия таблиц: `hay-xlsm-v1` (см. ФАЗА 2).
- Дата прогона: 2026-06-21.

## 1. Импорт документа: что дал ИИ-импортёр, а что — нет

`DossierExtractionAgent().extract(...)` (тот же ИИ-импорт, что доступен из API
`/api/import/document?use_ai=true`) корректно вытащил из `.docx`:

- название, ДЗО, подразделение, руководителя, цель должности (дословно);
- 4 области ответственности → `responsibilities`;
- 7 показателей эффективности → `kpis`;
- `headcount=11` (единственная количественная цифра масштаба в документе — годовых
  денежных показателей в этом ОД нет вообще).

Импортёр **честно оставил пустыми** 19 полей (`missing_fields`), включая
`key_results`, `decides_alone`/`requires_approval`/`recommends`, все денежные
показатели масштаба, `stakeholders`, `organizational_context`, `anchor_roles`,
`problem_cases`, `confirmed_by` — потому что этого текста в документе действительно
нет. Это не баг импортёра: используемый шаблон ОД КМГ (тот же `© 2015 Hay Group`
шаблон, что и в файле) **структурно не содержит** разбивку полномочий
«сам/согласует/рекомендует», финансовый масштаб и типовые кейсы Problem Solving —
хотя именно эти поля нужны методике Hay для уверенной оценки.

**Находка для процесса импорта:** для документов этого шаблона Gate 0 почти всегда
будет давать `NEEDS_CLARIFICATION` без ручной донормировки — это системное
свойство шаблона ОД, а не проблема парсера.

## 2. Что добавлено вручную перед оценкой (явно помечено)

Чтобы пройти Gate 0 и получить реальную оценку (а не `CANNOT_EVALUATE`), следующие
поля доопределены мной по тексту документа (не агентом, не импортёром) — каждое
прямо выводится из раздела «Основные показатели эффективности»/«Основные области
ответственности» документа, без догадок о фактах, которых в тексте нет:

- `key_results` (5 пунктов) — переформулированы из «Критериев эффективности и
  результативности» документа в форму результатов, а не процессов.
- `organizational_context` — собран из «Подчиняется» + «17 ПСП для взаимодействия».
- `authorities` (частично) — 2 пункта `decides_alone` и 2 `requires_approval`
  взяты из явных формулировок «организует контроль…» (сам) и «принимает участие в
  разработке специальных планов… для ликвидации сложных аварий» / «согласовывает
  материалы для… отвода» (согласование с комиссией/госорганами).

**Не восстановлено** (и не должно быть): денежный масштаб, `anchor_roles`,
`problem_cases`, `confirmed_by` — в документе действительно нет данных для этого.

## 3. Результат Gate 0

Статус: **`needs_clarification`** (не `cannot_evaluate` — критические блоки
закрыты после ручной донормировки из п.2). Предупреждения (рекомендуемые блоки):

| Блок | Почему |
|---|---|
| Якорные должности | В документе нет сопоставимых должностей для калибровки |
| Типовые кейсы (Problem Solving) | Документ не содержит структурированных кейсов — только общее упоминание аварийных планов |
| Лимиты (бюджет, закупки, штат) | ОД не специфицирует лимиты полномочий |
| Подтверждение руководителя/HR | В документе подпись `Утверждено` без даты — фактически черновик/шаблон |

Это реалистичный результат: документ — официальный, но по формату не покрывает
все поля, которые методика считает желательными для уверенной оценки.

## 5. QC и рекомендация системы

- `accountability_non_quantitative_policy`, `person_not_role`, `pay_independence`,
  `neutral_summary` — **PASS**, без замечаний.
- `high_complexity_few_cases` — **WARN**: сложность 4 при отсутствии 3+ типовых
  кейсов (см. таблицу выше, строка «Сложность»). Это правильное срабатывание
  правила из ФАЗЫ 4 — не ложный флаг.
- Итоговый статус: `needs_clarification`. Системная рекомендация: *«Предварительно
  зафиксировать Know-How F/III/2, Problem Solving E/4, Accountability F/N/V: 890
  баллов, грейд 21, профиль A1. Не утверждать до закрытия 1 замечания QC и
  калибровки спорных подфакторов»* — рекомендация системы текстуально совпадает
  с выводом ручного разбора выше (спорные подфакторы — Specialized Know-How F/E
  и Freedom to Act F/E).

## 6. Выводы для калибровки экспертов

1. **Уровни, которые методика подтверждает без оговорок:** Managerial Know-How
   III, Communication 2, Problem Area E, Magnitude N, Non-quantitative impact V.
2. **Уровни, требующие калибровки комитетом (граница не закрыта доказательствами
   агента):** Specialized Know-How (F против соседнего E) и Freedom to Act
   (F против соседнего E). В обоих случаях текст документа совместим с обоими
   соседними уровнями; решающего факта, разделяющего их, агент не привёл.
3. **Системная находка, не специфичная для этой роли:** несмотря на то, что
   системный промпт (`jeval/agent/prompt.py`) явно требует для каждого
   подфактора «выбранный уровень, ближайший соседний уровень, почему роль не
   ниже и не выше», реальный ответ `GroqAgent` (llama-3.3-70b) оставил
   `adjacent_level=null` и дал по 2 коротких пункта доказательств на фактор без
   сравнения с соседней ячейкой — то есть модель **не выполнила это требование
   промпта**. Это снижает аудируемость именно спорных уровней (п. 2) и не
   связано с правильностью движка/таблиц (баллы посчитаны детерминированно и
   верно). Стоит рассмотреть либо усиление промпта (явный tool-параметр
   `adjacent_level` как required, не optional), либо постобработку, требующую
   повторного запроса у модели при пустом `adjacent_level` для факторов с
   высокой неоднозначностью.
4. **Находка про формат документа:** шаблон ОД КМГ (Hay Group, 2015) не несёт
   разбивку полномочий (сам/согласует/рекомендует), финансовый масштаб и
   типовые кейсы Problem Solving — три поля, прямо нужные методике. Это
   объясняет, почему Gate 0 почти неизбежно даёт `needs_clarification` для
   документов этого формата без ручной донормировки, и не является дефектом
   парсера/Gate 0.

## Как повторить

```bash
cd backend && source .venv/bin/activate
python3 - <<'PY'
from datetime import date
from jeval.domain.models import ApprovalItem, Authorities, JobDossier, Reporting, Scope
from jeval.orchestrator import JobEvaluator
from jeval.agent.groq_agent import GroqAgent  # требует GROQ_API_KEY в .env

dossier = JobDossier(
    name="Директор департамента бурения и капитального ремонта скважин",
    dzo="АО «Озенмунайгаз»",
    purpose="...",                # см. текст документа
    key_results=[...],            # см. раздел 2 этого файла
    responsibilities=[...],
    kpis=[...],
    reporting=Reporting(manager="Заместитель генерального директора по геологии и разработке"),
    authorities=Authorities(decides_alone=[...], requires_approval=[ApprovalItem(item="...", approver="...")]),
    scope=Scope(headcount=11, source="ОД, раздел «Количественные показатели масштаба деятельности»"),
    organizational_context="...",
)
print(JobEvaluator(agent=GroqAgent()).evaluate(dossier).model_dump_json(indent=2))
PY
```

Полный текст исходного документа извлекается через
`jeval.importer.extract_docx_text` (используется и в `/api/import/document`).
