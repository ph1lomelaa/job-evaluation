# Архитектура

## Принцип: LLM выбирает уровни, код считает баллы

Главное архитектурное решение — разделить недетерминированную и детерминированную
части:

- **Агент (Claude)** читает JE-досье и возвращает только *уровни* факторов
  (A–H, 1–5, P/S/C/R), доказательства и уверенность.
- **Движок (код)** делает подстановку по таблицам Hay → баллы → профиль → грейд.
- **Импорт** сохраняет provenance: имя файла, размер, хеш, MIME и карту
  `field_sources`, чтобы контекст можно было проследить от документа до оценки.

Это требование методики («баллы только через разрешённые таблицы») и даёт
воспроизводимость: при одних и тех же уровнях баллы и грейд всегда одинаковы.

## Конвейер

```
JobDossier
   │
   ▼  jeval/gate.py        Gate 0: полнота данных. FAIL → стоп (нет оценки)
   ▼  jeval/agent          Claude → уровни факторов + доказательства
   ▼  jeval/scoring        таблицы Hay → баллы → профиль → грейд
   ▼  jeval/qc.py          QC-флаги (персонализация, оплата, связки факторов)
   ▼  jeval/orchestrator   сборка в Evaluation (карточка для комитета)
   ▼  jeval/store.py       сохранение (SQLite / in-memory)
   ▼  jeval/api            HTTP (FastAPI)
```

## Компоненты

| Слой | Модуль | Зависит от сети |
|------|--------|-----------------|
| Модели | `jeval/domain` | нет |
| Расчёт | `jeval/scoring` | нет |
| Gate 0 | `jeval/gate.py` | нет |
| QC | `jeval/qc.py` | нет |
| Агент | `jeval/agent` | **да** (Anthropic API) |
| Оркестратор | `jeval/orchestrator.py` | через агент (инъектируется) |
| Хранилище | `jeval/store.py` | нет |
| HTTP | `jeval/api` | нет |

Агент инъектируется в `JobEvaluator`, поэтому весь конвейер тестируется офлайн
с фейковым агентом (см. `tests/` и `scripts/demo.py`).

## Точность таблиц

`jeval/scoring/tables.py` содержит флаг `TABLES_VERIFIED`. Грейд-матрица и матрица
Problem Solving % — точные. Know-How и Accountability — предварительные
(аддитивно-шаговая модель Hay), подлежат сверке с официальным чартом Korn Ferry.

## Дальнейшее развитие

- Сверка и фиксация таблиц Know-How / Accountability (`TABLES_VERIFIED = True`).
- Калибровка с якорными должностями (сравнение оценок).
- UI по `frontend/FIGMA_PROMPT.md`.
- Экспорт карточки в Word/PDF.

## Multi-tenant контур

Пользователь — глобальная сущность. Компания — tenant, а связь пользователя с
компанией хранится в `company_memberships`. Это позволяет одному аккаунту работать
с несколькими компаниями без дублирования логина и безопасно переключать контекст.

```text
users ──< company_memberships >── companies
                                      │
                                      ├──< positions ──< evaluations
                                      ├──< public_forms
                                      └──< audit_events

users ──< sessions (только SHA-256 хеш токена)
```

### Инварианты изоляции

1. Каждый закрытый запрос содержит bearer-сессию или session-cookie и `X-Company-ID`.
2. Backend проверяет активную membership, а не доверяет заголовку клиента.
3. Каждая выборка позиции, оценки или формы включает фильтр `company_id`.
4. Объект другого tenant возвращает `404`, а не `403`.
5. Публичная форма сама хранит `company_id`; при отправке клиент его не задаёт.

### Почему hybrid relational + JSON

Поля, по которым строятся ограничения и выборки (`company_id`, `position_id`,
статусы, грейд, баллы, даты), вынесены в обычные колонки с индексами. Полное
вложенное JE-досье и карточка оценки сохраняются в JSON. Это даёт строгую tenant-
изоляцию и быстрые списки, но не требует миграции схемы при каждом изменении
структуры методики Hay.

Runtime разработки использует SQLite в WAL-режиме. Целевой production adapter —
PostgreSQL с теми же таблицами и ограничениями; при нескольких backend-инстансах
SQLite использовать нельзя.

### Миграция существующих данных

`SqliteStore` добавляет новые колонки и таблицы без удаления старых JSON-документов,
затем backfill-ит индексируемые поля. При создании самой первой компании записи
без tenant автоматически привязываются к ней — это сохраняет данные прежней
однопользовательской версии.
