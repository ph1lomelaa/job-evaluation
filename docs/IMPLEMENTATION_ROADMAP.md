# План внедрения методики и трассируемости

## Корпоративные решения

1. Доход, выручка и денежные диапазоны не участвуют в оценке Accountability.
2. Для каждой новой оценки применяется только ветка `Magnitude N`.
3. При `N` выбирается не тип влияния `R/C/S/P`, а объединённый
   неколичественный уровень воздействия `I–VI`.
4. AI предлагает уровни, но не утверждает грейд. Баллы считает только
   детерминированный движок по функциям XLSM.

## Когда ставится «+» или «−»

Модификатор отражает положение между соседними ячейками таблицы, а не качество
досье и не уверенность эксперта.

- `+`: требования роли устойчиво выше полного описания выбранной базовой ячейки,
  но доказательств недостаточно для следующей полной ячейки.
- `−`: выбранная ячейка достигается только по нижней границе; факты сильнее
  предыдущей ячейки, но не полностью соответствуют базовой.
- `0`: описание одной ячейки подтверждено без документированной пограничности.
- Несколько плюсов или минусов внутри одной факторной таблицы не суммируются:
  итоговый сдвиг ограничен одним шагом.
- Для `Know-How` и `Accountability` `+`/`−` сдвигает результат на один шаг ряда Hay.
- Для `Problem Solving` XLSM обрабатывает агрегированный `+` как переход на
  следующий процентный шаг. Агрегированный `−` помечает нижнюю границу, но не
  уменьшает базовый процент; это воспроизведено в движке.
- Для любого ненулевого модификатора обязательны `modifier_reason` и
  `adjacent_level`. Их отсутствие создаёт QC-предупреждение.

## Неколичественная ветка N

| Уровень | Смысл | Что должно быть в доказательствах |
|---|---|---|
| I | Вспомогательное | Разовая услуга или отдельная операция для другой роли |
| II | Поддерживающее | Информационная/административная поддержка внутри подразделения |
| III | Операционное | Несколько подразделений или поддержка ключевого процесса/оборудования |
| IV | Аналитическое | Специализированная аналитика, диагностика, консультации или критичная сложная система |
| V | Направляющее | Руководство направлениями, командами, проектами или системные решения |
| VI | Воздействующее | Несколько связанных команд, стратегические программы или политика всей организации |

3. **Выполнено:** закрепить правило «финансовый доход не используется» в промпте и QC.
4. **Выполнено:** добавить `NonQuantitativeImpact I–VI` в доменную модель и API.
5. **Выполнено:** реализовать расчёт `FINALITE` для `N/I–VI` и контрольные тесты.
6. **Выполнено:** формализовать условия модификаторов и обязательное обоснование границы.

17. Для каждого уровня показывать соседние уровни и объяснение выбора между ними.

## Источники правил

- `Калькулятор Hay Group.xlsm`: функции `COMP`, `IC`, `PTSIC`, `FINALITE`, ряд Hay и таблица грейдов.
- `Калькулятор Hay Group.md`: текстовое описание структуры калькулятора.
- `Постановочные таблицы Hay Group.md/.pdf`: дескрипторы уровней и отдельные количественная/неколичественная ветки.
- `Руководство_для_эксперта_по_оценке.md/.pdf`: порядок экспертного выбора, пограничность и контроль качества.
- `Тренинг_оценка день 1 — раздатка.md/.pdf`: логика факторов и калибровка оценок.
- `Шаблон_ведомости_оценки_должностей.docx/.md`: требуемая форма итогового реестра.
- `1. ОД Директора ДБиКРС.docx`: реальный формат входного досье для проверки импорта.
