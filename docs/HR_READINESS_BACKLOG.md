### 1. Починить AI-импорт `.docx` без ключа / при сбое провайдера — ✅ исправлено 23.06.2026

**Проблема.** Дашборд предлагает HR главную фичу — перетащить существующую
должностную инструкцию (.docx) и получить черновик. Я воспроизвёл: загрузка
настоящего файла `1. ОД Директора ДБиКРС.docx` через UI **падает** с "ИИ-импорт
временно недоступен. Попробуйте позже или без ИИ." — и дальше тупик, кнопки
"без ИИ" нет.

**Причина.** [`DashboardPage.tsx:213`](../frontend/src/pages/DashboardPage.tsx#L213)
жёстко вызывает `api.importDocument(file, true)` — форсирует `use_ai=true`.
Backend же по умолчанию (`jeval_import_use_ai=False`,
[`config.py`](../backend/jeval/config.py)) и без AI **реально работает**: я
прогнал тот же файл через `use_ai=false` и получил корректно извлечённые
name/ДЗО/подразделение/цель/3 ключевых результата/17 обязанностей/10 KPI/
численность (11 чел.) с явным списком, чего не хватает.

**Что сделано.** [`DashboardPage.tsx`](../frontend/src/pages/DashboardPage.tsx)
теперь сначала пробует `use_ai=true`; если backend отвечает `503` (провайдер не
настроен/недоступен), автоматически повторяет запрос с `use_ai=false` —
heuristic-экстрактор уже умел это делать, просто не было пути до него с фронта.
При срабатывании fallback пользователь видит не ошибку, а информационный
баннер (новый компонент [`NoticeBanner`](../frontend/src/components/ui.tsx))
на карточке оценки: "Черновик создан без ИИ (по ключевым словам документа) —
точность ниже. Проверьте все поля перед запуском оценки." Прочие ошибки (не
503 — например, неверный формат файла) показываются как раньше, без fallback.

Проверено вручную: backend без `OPENAI_API_KEY`/`ANTHROPIC_API_KEY`, загрузка
настоящего `1. ОД Директора ДБиКРС.docx` через UI — черновик создаётся,
баннер отображается, тупика больше нет.

**Не сделано (можно вернуться отдельно):** чекбокс "Использовать ИИ" для
случаев, когда ключ настроен, но HR сознательно хочет более быстрый
heuristic-режим — сейчас это решает только автоматический fallback по ошибке.

### 2. Убрать технические имена переменных из текста для HR — ✅ исправлено 23.06.2026

**Проблема.** В карточке оценки при офлайн-режиме (FakeAgent) текст обоснования
буквально показывает HR имена переменных окружения:
*"Для точной оценки укажите `ANTHROPIC_API_KEY`/`GROQ_API_KEY`/`OPENAI_API_KEY`
и `JEVAL_AGENT_PROVIDER` в `.env`"*. Сам баннер "ТЕСТОВЫЕ ДАННЫЕ — НЕ ДЛЯ
КОМИТЕТА" — это правильная и важная защита, текст про `.env` — нет: HR
комплаенс-офицер крупной компании не должен видеть имена переменных окружения.

**Что сделано.** В [`jeval/agent/fake.py`](../backend/jeval/agent/fake.py)
(текст `reasoning`, который идёт в каждую офлайн-оценку) и в хардкоженном
баннере на [`EvaluationCardPage.tsx`](../frontend/src/pages/EvaluationCardPage.tsx#L221-L225)
упоминания `ANTHROPIC_API_KEY`/`GROQ_API_KEY`/`OPENAI_API_KEY`/`JEVAL_AGENT_PROVIDER`
и внутреннее имя класса `FakeAgent` заменены на "обратитесь к администратору
системы, чтобы подключить реальный AI-агент". Сам факт "это тест, не для
комитета" — не трогали, он и так работает правильно. Backend: все 213 тестов
прошли (ни один не был завязан на старый текст). Frontend: `tsc --noEmit` чистый.

### 3. PDF-экспорт карточки оценки под официальный шаблон — ✅ исправлено 23.06.2026

**Проблема.** Единственный способ "выгрузить" карточку — `window.print()`
([`EvaluationCardPage.tsx:199`](../frontend/src/pages/EvaluationCardPage.tsx#L199)),
то есть Ctrl+P в браузере. В корне репозитория уже лежит официальный
`Шаблон_ведомости_оценки_должностей.docx` — но автогенерации карточки в этот
формат нет. Комитету крупной компании нужен подписываемый, архивируемый
документ, а не распечатка веб-страницы.

**Что сделано.** Новый эндпоинт
[`GET /api/evaluations/{id}/export.pdf`](../backend/jeval/api/routers/evaluations.py)
рендерит PDF через ReportLab
([`jeval/export/pdf.py`](../backend/jeval/export/pdf.py)): позиция, статус,
таблица грейд/баллы/профиль/надёжность, факторная таблица с кодами и баллами,
формула, подробное объяснение расчёта, резюме должности, обоснование,
QC-флаги (FAIL/WARN со списком) и рекомендация для комитета. Если
`is_test_data` — PDF открывается явным красным баннером "ТЕСТОВЫЕ ДАННЫЕ — НЕ
ДЛЯ ОЦЕНОЧНОГО КОМИТЕТА" с тем же текстом, что на экране (см. задачу 2).

Кириллица — основной язык контента, а встроенные шрифты ReportLab её не
содержат. Резолвер [`jeval/export/fonts.py`](../backend/jeval/export/fonts.py)
ищет TTF с кириллицей по списку известных путей (без вендоринга бинарника в
репозиторий): в контейнере это `fonts-dejavu-core`, добавленный в
[`backend/Dockerfile`](../backend/Dockerfile); локально — уже установленные в
ОС шрифты, с возможностью явно указать путь через `JEVAL_PDF_FONT_PATH`.

На фронте рядом с существующей кнопкой "Печать" (бывшая "PDF" — переименована,
чтобы не путать с настоящим экспортом) добавлена кнопка "Экспорт в PDF"
([`EvaluationCardPage.tsx`](../frontend/src/pages/EvaluationCardPage.tsx),
`api.exportEvaluationPdf` в [`lib/api.ts`](../frontend/src/lib/api.ts)) — в
отличие от "Печать", она не блокируется на тестовых данных, потому что PDF и
так явно помечает их баннером.

Проверено сквозь реальный браузер (Playwright + Chrome): регистрация → создание
должности с кириллицей в названии (включая `&` и кавычки) → запуск оценки →
клик "Экспорт в PDF" → скачанный файл — валидный PDF, `pdftotext` подтвердил
кириллицу без артефактов и наличие всех секций. В процессе найден и исправлен
баг: код подфактора Problem Solving показывал `ProblemComplexity.VARIABLE`
вместо `3` (этот enum — обычный `int, Enum`, а не проектный `StrEnum`, поэтому
f-string без `.value` отдавал имя члена, а не значение).

**Не относится к этой задаче, но всплыло при проверке:** локальный файл
`backend/jeval.db` от старых запусков может содержать схему `audit_events`,
не совместимую с текущим `CREATE TABLE IF NOT EXISTS` (ловится как
`sqlite3.IntegrityError: datatype mismatch` при записи аудит-лога, ломает
регистрацию/логин с криптичной CORS-ошибкой в браузере — на самом деле сервер
просто роняет соединение без ответа). Не трогал этот файл намеренно — в нём
может быть тестовые данные с прошлого ручного аудита. Если столкнётесь с этим
после `git pull`, скорее всего проще пересоздать локальную `jeval.db`, чем
писать миграцию для одной таблицы аудита в dev-окружении.

**Готово, когда:** ~~из карточки одной кнопкой скачивается PDF, который можно
подписать и подшить в досье комитета.~~ Готово.

### 4. Excel/CSV-выгрузка реестра должностей — ✅ исправлено 23.06.2026

**Проблема.** Нет способа выгрузить список должностей (грейды, профили,
статусы) для отчёта руководству, выгрузки в HRIS/payroll или архива. Только
просмотр построчно на дашборде/в "Ведомости".

**Что сделано.** Реализовано на клиенте, без нового backend-эндпоинта: баллы,
коды факторов и профиль уже посчитаны и отрендерены на странице — выгружаем
ровно то же самое, не дублируя логику подсчёта в Python. Новая утилита
[`lib/csv.ts`](../frontend/src/lib/csv.ts) (RFC 4180, `;` как разделитель —
в ru-RU локали Excel запятая зарезервирована под десятичный разделитель, BOM
для кириллицы) + кнопка "Экспорт в CSV":
- на [`DashboardPage.tsx`](../frontend/src/pages/DashboardPage.tsx) — простой
  реестр (должность/ДЗО/функция/статус/грейд/уверенность), с учётом активных
  фильтров (поиск, подразделение, статус);
- на [`AssessmentSheetPage.tsx`](../frontend/src/pages/AssessmentSheetPage.tsx) —
  полная факторная разбивка (коды Know-How/Problem Solving/Accountability,
  модификаторы +/−, баллы, профиль, грейд, статус).

Проверено вручную: создана и оценена тестовая должность, оба экспорта скачаны
и открыты — заголовки и кириллица корректны, столбцы размечены верно.

### 5. Пагинация и серверная фильтрация списка должностей

[`positions.py`](../backend/jeval/api/routers/positions.py), ни в
[`api.ts`](../frontend/src/lib/api.ts). При росте до сотен/тысяч должностей
(реалистично для масштаба крупной компании типа КМГ с множеством ДЗО) дашборд
будет каждый раз грузить и фильтровать всё на клиенте.

**Что сделать.**
- Добавить `limit`/`offset` (или cursor) и серверную фильтрацию по
  статусу/подразделению/поиску в `GET /api/positions`.
- Перевести `DashboardPage.tsx`/`AssessmentSheetPage.tsx` на постраничную
  загрузку или infinite scroll вместо `useFetch` всего массива сразу.

**Готово, когда:** при 1000+ должностей в компании дашборд открывается без
заметной задержки и без загрузки всего реестра на клиент.

### 6. Импорт ДИ из PDF и сканов, не только `.docx`

**Проблема.** Импорт документов жёстко ограничен `.docx`
([`positions.py:173`](../backend/jeval/api/routers/positions.py#L173):
`if not safe_name.lower().endswith(".docx")`). Архив должностных инструкций
крупной компании почти гарантированно смешанный: часть в Word, часть в PDF,
часть — сканы старых приказов.

**Что сделать.**
- Добавить извлечение текста из PDF (текстовый слой — `pdfplumber`/`pypdf`).
- Для сканов — OCR-этап (например, `pytesseract`) с понижением `confidence`
  в `import_metadata`, как уже сделано для heuristic-режима.
- Расширить `_ALLOWED_DOCUMENT_SUFFIXES`/проверку расширения в
  `import_document` и явно показать пользователю, какие форматы поддержаны.

**Готово, когда:** HR может загрузить и `.docx`, и `.pdf` ДИ и получить черновик
с тем же уровнем прозрачности (`extraction_method`, `confidence`,
`missing_fields`).

### 7. Зафиксировать единый host для локальной разработки

**Проблема (не прод-баг, только локальная разработка).** Дефолтный
[`frontend/.env`](../frontend/.env) указывает `VITE_API_URL=http://127.0.0.1:8001`,
а Vite по умолчанию поднимается на `localhost:5173`. Браузер считает
`localhost` и `127.0.0.1` разными сайтами → cookie сессии с `SameSite=Lax` не
долетают при XHR → после регистрации/входа все последующие запросы (создание
компании и т.д.) падают с 401, хотя сама регистрация прошла. В реальном
деплое (docker-compose + nginx, см. [`docker-compose.yml`](../docker-compose.yml))
фронт и API на одном origin — там этой проблемы нет.

**Что сделать.** В `frontend/package.json` `scripts.dev` добавить
`vite --host 127.0.0.1`, либо одной строкой в README указать "открывайте
`http://127.0.0.1:5173`, не `localhost:5173`".

**Готово, когда:** разработчик, впервые поднявший проект по README, не
натыкается на необъяснимые 401 после регистрации.
